# YiLa 硬件接口文档

本文档只记录“和硬件设备交互”的接口、协议和扩展约定，方便以后做 Web、App、桌面端、Home Assistant、本地 API 或调试工具时复用。

参考来源：

- 当前项目：`packages/yila-core`、`apps/web/src/yila-web-ble.ts`
- 原项目：`JustaSage/OpenYiLa`，源码版本 `1.1.2`

## 功能清单

| 功能 | 当前项目 | 原项目 | 说明 |
| --- | --- | --- | --- |
| 扫描设备 | 部分实现 | 已实现 | Web 端通过浏览器蓝牙选择器；原项目可扫描附近设备、RSSI 和广播电量 |
| 连接设备 | 已实现 | 已实现 | 连接 GATT，获取 NUS TX/RX |
| 断开连接 | 已实现 | 已实现 | 断开前停止 RX notification，再断开 GATT |
| 开门 | 已实现 | 已实现 | 发送 `A:OPEN` 加密指令 |
| 修改密码 | 未接入 | 已实现 | 发送 `A:PW` 加密指令 |

## BLE 参数

| 名称 | 值 |
| --- | --- |
| 默认设备名称前缀 | `YILA` |
| Service UUID | `6e400001-b5a3-f393-e0a9-e50e24dcca9e` |
| TX Characteristic | `6e400002-b5a3-f393-e0a9-e50e24dcca9e` |
| RX Characteristic | `6e400003-b5a3-f393-e0a9-e50e24dcca9e` |
| CCCD UUID | `00002902-0000-1000-8000-00805f9b34fb` |

TX/RX 方向：

```text
TX: phone/app/web -> device，用于写入加密指令
RX: device -> phone/app/web，用于接收 notification 响应
```

连接流程：

```text
扫描或选择 BLE 设备
  -> 连接 GATT
  -> 获取 NUS Service
  -> 获取 TX / RX Characteristic
  -> RX 开启 notification
  -> TX 写入硬件指令
  -> RX 等待响应
```

字段映射：

## 扫描设备

原项目接口：

```ts
type ScannedDevice = {
  address: string;
  name: string;
  rssi: number;
  battery_level?: number;
};

scanDevices(timeoutSeconds = 6.0): Promise<ScannedDevice[]>;
```

行为：

- 使用 BLE 扫描附近设备。
- 设备名称包含 `YILA` 即视为 YiLa 设备。
- 返回 `address`、`name`、`rssi`、`battery_level`。
- 如果广播数据里能解析出电量，则返回 `battery_level`。

当前 Web 状态：

- Web Bluetooth 不能像原生 BLE 那样自由后台扫描。
- 当前使用 `navigator.bluetooth.requestDevice` 让用户选择设备。
- 当前没有读取 RSSI 和广播电量。

## 连接设备

```ts
type DeviceRequestOptions = {
  namePrefix?: string;
  deviceId?: string;
  address?: string;
  connectTimeoutMs?: number;
};

type ConnectionInfo = {
  deviceName: string;
  deviceId: string;
  address?: string;
};

connect(options?: DeviceRequestOptions): Promise<ConnectionInfo>;
```

说明：

- `namePrefix` 为空时使用 `YILA`。
- 原项目使用 BLE `address` 连接。
- Web Bluetooth 使用浏览器给出的 `device.id`，不一定等于 MAC 地址。
- Web 端必须由用户点击触发蓝牙选择器。
- 连接成功后可保存设备 ID、名称、地址、RSSI、电量和设备级设置，但不得保存真实密码。
- 当前 Web 不手写 CCCD；`startNotifications()` 由浏览器负责写 CCCD。

## 断开连接

```ts
disconnect(): void | Promise<void>;
```

说明：

- 当前 Web 先调用 `rx.stopNotifications()`，再断开 GATT 并清空 TX/RX。
- 原项目断开前会尝试 `stop_notify(NUS_RX_CHAR_UUID)`。
- 断开后必须重新连接才能继续写入硬件指令。
- 当前 Web 不手写 CCCD；`stopNotifications()` 由浏览器负责关闭 CCCD。

## 开门

```ts
type UnlockOptions = {
  password: string;
  openTimeMs: number;
  waitTimeMs: number;
  closeTimeMs: number;
  reverse: boolean;
  timestampSeconds?: number;
};

open(options: UnlockOptions, timeoutMs = 6000): Promise<DeviceResponse>;
```

参数说明：

| 参数 | 类型 | 说明 |
| --- | --- | --- |
| `password` | `string` | 6 位数字设备密码，只在运行时输入 |
| `openTimeMs` | `number` | 开锁动作时间 |
| `waitTimeMs` | `number` | 开锁后等待时间 |
| `closeTimeMs` | `number` | 关锁动作时间 |
| `reverse` | `boolean` | 是否反向开锁 |
| `timestampSeconds` | `number` | 可选；默认当前 Unix 秒，主要用于测试 |
| `timeoutMs` | `number` | 等待硬件响应超时 |

默认值差异：

| 来源 | open | wait | close | reverse/attribute |
| --- | --- | --- | --- | --- |
| 当前 Web 默认值 | `1500` | `1000` | `800` | `false` |
| 原项目 CLI / MCP 默认值 | `500` | `500` | `500` | `false` |
| 原项目 `YilaDevice` 模型默认值 | `650` | `2000` | `650` | `0` |
| 原项目 README 参数表 | `500` | `500` | `500` | `0` |

后续新增平台时，应以设备实测效果为准，并允许每个设备独立保存时序。

开门流程：

```text
校验 6 位数字密码
  -> buildOpenCommand
  -> TX write without response
  -> RX 等待 notification
  -> parseDeviceResponse
  -> 返回 DeviceResponse
```

## 开门指令格式

```ts
buildOpenCommand(options: UnlockOptions): Uint8Array;
```

payload：

```text
A:OPEN;P:{direction} {openTimeMs},{waitTimeMs},{closeTimeMs};
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `direction` | `+` 表示正常方向，`-` 表示反向 |
| `openTimeMs` | 开锁动作时间 |
| `waitTimeMs` | 等待时间 |
| `closeTimeMs` | 关锁动作时间 |

完整明文：

```text
{timestampSeconds}{passwordKey}A:OPEN;P:{direction} {openTimeMs},{waitTimeMs},{closeTimeMs};
```

其中：

```text
passwordKey = MD5(password).slice(8, 24)
```

## 修改密码

原项目已实现，当前 Web 未接入。

```ts
type ChangePasswordOptions = {
  oldPassword: string;
  newPassword: string;
};

changePassword(options: ChangePasswordOptions, timeoutMs = 6000): Promise<DeviceResponse>;
```

参数说明：

| 参数 | 类型 | 说明 |
| --- | --- | --- |
| `oldPassword` | `string` | 当前 6 位数字密码 |
| `newPassword` | `string` | 新 6 位数字密码 |

payload：

```text
A:PW;P:{newPasswordKey};
```

完整明文：

```text
{timestampSeconds}{oldPasswordKey}A:PW;P:{newPasswordKey};
```

其中：

```text
oldPasswordKey = MD5(oldPassword).slice(8, 24)
newPasswordKey = MD5(newPassword).slice(8, 24)
```

注意：

- 修改密码必须使用旧密码生成明文头部。
- 新密码写入的是 `newPasswordKey`，不是明文新密码。
- 新旧密码都必须是 6 位数字。
- 新旧密码不得保存到配置、数据库、日志或测试快照。

## 初始化密码

原项目协议层有：

```ts
buildInitPasswordCommand(oldPassword: string, newPassword: string): Uint8Array;
```

当前已知行为：

- 指令格式与修改密码相同。
- 原项目源码中 `build_init_password_command` 直接调用 `build_change_password_command`。
- 当前 Web 未接入。

## 加密规则

所有已知写入指令使用同一套外层加密：

```text
AES-128-ECB
Zero Padding
Outer Key: Fx4k6AWivOsLE4NI
```

构造流程：

```text
1. 派生 passwordKey: MD5(password).slice(8, 24)
2. 拼接明文: "{unix_timestamp}{passwordKey}{payload}"
3. UTF-8 编码
4. Zero Padding 到 16 字节边界
5. AES-128-ECB 加密
6. 写入 NUS TX
```

安全要求：

- 文档记录固定 key 是为了协议实现。
- 用户真实密码必须运行时输入。
- 日志不得输出真实密码、明文指令、派生 key 或可复原密码的中间值。

## 响应解析

```ts
parseDeviceResponse(data: Uint8Array): DeviceResponse;
```

已支持响应：

| 响应内容 | 结果 |
| --- | --- |
| 空响应 | `success: false`, `message: "Empty response"` |
| 包含 `OK` | `success: true`, `message: "OK"` |
| 包含 `ERROR` | `success: false`, `message: "Password error"` |
| 包含 `FAIL` | `success: false`, `message: "FAIL"` |
| 十六进制包含 `4F4B` | 按 `OK` 处理 |
| 十六进制包含 `4552524F52` | 按 `ERROR` 处理 |
| 单字节 `1..5` | 解析为电量等级 |
| 文本包含 `BAT/BATT/BATTERY/PWR/POWER` + `1..5` | 解析为电量等级 |
| 未识别 | `success: false`, `message: "Unknown response"` |

电量约定：

```ts
batteryLevel?: 1 | 2 | 3 | 4 | 5;
```

当前代码里类型是 `number`，业务层使用时应按 `1..5` 显示。

## 广播电量解析

原项目已实现，当前 Web 未接入。

```ts
parseAdvertisementBattery(manufacturerData: Uint8Array): number | undefined;
```

算法：

```text
先按完整 advertisement data block 解析
  -> 如果 block 最后一个字节在 1..5，认为是电量等级
否则按 BLE 库返回的 manufacturer data value 解析
  -> 如果整段数据最后一个字节在 1..5，认为是电量等级
```

说明：

- 原项目按完整 AD block 解析。
- 实测 bleak 返回的 `manufacturer_data` value 可能已经剥离了 AD length/type，因此需要 fallback 到“最后一个字节”判断。

返回：

| 返回值 | 说明 |
| --- | --- |
| `1` | 低电量 |
| `2` | 约 25% |
| `3` | 约 50% |
| `4` | 约 75% |
| `5` | 满电 |
| `undefined` | 未解析到电量 |

注意：

- Web Bluetooth 对广播数据暴露有限，不一定能实现和原生 App 相同的扫描电量。
- Android、iOS、桌面端和本地服务端更适合实现该接口。

## 设备记录建议

硬件设备记录应该由客户端或本地数据库保存，不属于协议核心。

```ts
type DeviceRecord = {
  id: string;
  name: string;
  address?: string;
  rssi?: number;
  batteryLevel?: number;
  settings: {
    openTimeMs: number;
    waitTimeMs: number;
    closeTimeMs: number;
    reverse: boolean;
  };
  createdAt: number;
  updatedAt: number;
  lastConnectedAt?: number;
  lastOpenedAt?: number;
};
```

保存规则：

- 每个设备保存独立设置。
- 原生端应保存真实 BLE 地址。
- Web 端保存浏览器提供的 `device.id`。
- 扫描拿到的 `rssi` 和 `batteryLevel` 可以保存为最近一次扫描值。
- 不保存真实密码。
- Web 端当前使用浏览器本地存储。
- App 端建议使用 SQLite。
- 服务端部署版可使用 SQLite 或 PostgreSQL。

## 适配层约定

以后多端复用时，建议所有平台实现同一组硬件接口：

```ts
interface YilaHardwareClient {
  scan(options?: { namePrefix?: string; timeoutSeconds?: number }): Promise<ScannedDevice[]>;
  connect(options?: { namePrefix?: string; deviceId?: string; address?: string }): Promise<ConnectionInfo>;
  disconnect(): void | Promise<void>;
  open(options: UnlockOptions, timeoutMs?: number): Promise<DeviceResponse>;
  changePassword?(options: ChangePasswordOptions, timeoutMs?: number): Promise<DeviceResponse>;
  initPassword?(options: ChangePasswordOptions, timeoutMs?: number): Promise<DeviceResponse>;
  parseAdvertisementBattery?(manufacturerData: Uint8Array): number | undefined;
}
```

平台对应关系：

| 平台 | 蓝牙实现 | 数据保存 |
| --- | --- | --- |
| Web | Web Bluetooth | IndexedDB 或 localStorage |
| Android | 原生 BLE | SQLite |
| iOS | CoreBluetooth | SQLite |
| 桌面端 | 系统 BLE / 插件 | SQLite |
| 本地 API 服务 | 系统 BLE | SQLite / PostgreSQL |

## 扩展功能预留

新增硬件功能时，按下面格式补充：

```ts
type HardwareCommandName = "OPEN" | "PASSWORD" | "INIT_PASSWORD" | "STATUS" | "CONFIG";
```

建议新增文件结构：

```text
packages/yila-core/
  src/
    index.ts
    commands/
      open.ts
      password.ts
      status.ts
      config.ts
    advertisement/
      battery.ts
    responses/
      parse.ts
```

每个新功能必须记录：

- 指令名称
- 明文 payload
- 完整明文结构
- 加密方式
- 写入特征
- 响应格式
- 超时策略
- 是否需要密码
- 是否会修改设备状态

## 错误处理约定

| 场景 | 建议提示 |
| --- | --- |
| 浏览器不支持 Web Bluetooth | 当前浏览器不支持蓝牙 |
| 用户取消蓝牙选择 | 已取消选择设备 |
| GATT 不可用 | 蓝牙连接不可用 |
| 设备不支持 NUS | 不是支持的 YiLa 设备 |
| 未连接就开门 | 设备未连接 |
| 密码格式错误 | 密码必须是 6 位数字 |
| 响应超时 | 设备响应超时 |
| 密码错误 | 开门失败：密码错误 |
| 修改密码失败 | 修改密码失败：设备拒绝或密码错误 |
| 未知响应 | 开门失败：未知响应 |

## 安全约定

- 真实密码不得写入源码、配置文件、文档示例和测试快照。
- 密码只允许在开门或修改密码时由用户输入。
- 日志不得输出密码、明文指令、派生 key 或可复原密码的中间值。
- 修改密码必须二次确认。
- 如果以后支持远程 API，开门和修改密码接口必须加认证和访问控制。
- 如果以后支持保存密码，必须先设计本地加密和用户授权策略。
